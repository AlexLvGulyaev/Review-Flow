# Session log: Cursor Task 013 — UI milestone client/company layouts

## Исходный промпт (полный текст)

# Cursor Task 013 — UI milestone: разделение клиентского сайта и рабочего пространства компании

## Контекст

Проект: `review-flow`.

После Task 012 в документации зафиксировано разделение интерфейсов:

- клиентский контур — публичный сайт вымышленной компании;
- контур компании — внутреннее рабочее пространство оператора и администратора.

Сейчас текущий frontend всё ещё выглядит как единый служебный тренажёр:
общая шапка, переключатель роли, ссылки в одном стиле.

Задача этого milestone:
реализовать визуальное и навигационное разделение интерфейсов без изменения backend pipeline.

---

# Главная цель

Сделать два разных интерфейсных мира:

Клиентский сайт компании:

- `/`
- `/review`
- `/review/status/:reviewId`

Рабочее пространство компании:

- `/company`
- `/operator/reviews`
- `/prompts`
- `/evaluation`
- `/analytics`
- `/logs`
- `/settings/ai-providers`
- `/admin/*`

Клиентский сайт должен выглядеть как внешний сайт компании, куда пользователь приходит оставить отзыв.

Контур компании должен выглядеть как внутренний рабочий интерфейс сотрудников.

---

# Визуальный ориентир

Использовать как ориентир ранее подготовленную UI-картинку docs/ui_reference_client_company_contours_v1.png :

- светлый клиентский сайт;
- отдельный header/footer;
- крупный hero-блок;
- CTA «Оставить отзыв»;
- аккуратная форма отзыва;
- тёмная или контрастная боковая навигация для внутреннего рабочего пространства компании;
- плотные таблицы и карточки в операторском/админском контуре.


Важно:
картинка является визуальным ориентиром, а не точной экранной спецификацией.

Не копировать выдуманные метрики или лишние разделы, которых нет в проекте.

# 1. Реализовать ClientLayout

Создать отдельную оболочку клиентского сайта.

## Страницы

### `/`

Новая главная страница вымышленной компании.

Рабочее название бренда можно использовать:

Northline Market

Если в проекте уже есть другое имя — использовать существующее.

Главная страница должна содержать:

- логотип/название компании;
- верхнее меню:
  - Главная;
  - Доставка и оплата;
  - Поддержка;
  - Оставить отзыв;
- hero-блок:
  - короткий текст о сервисе;
  - кнопка «Оставить отзыв»;
  - кнопка «Проверить статус обращения»;
- 3–4 карточки преимуществ:
  - Быстрая доставка;
  - Проверенное качество;
  - Поддержка клиентов;
  - Безопасная оплата;
- footer с тестовыми контактами.

Это статический демонстрационный сайт, без реального каталога товаров.

---

### `/review`

Переделать форму отзыва под клиентский сайт.

Требования:

- убрать ощущение служебной формы;
- использовать ClientLayout;
- не показывать переключатель роли;
- не показывать ссылки на operator/admin;
- поле `product_area` заменить на понятное поле:

Выберите тему обращения

Варианты:

- Доставка
- Качество товара
- Оплата
- Возврат
- Поддержка
- Сайт / приложение
- Другое

Для backend можно пока маппить выбранную тему в существующее поле `product_area`.

Если нужно, добавить небольшой frontend mapping:

| UI значение | product_area |
|---|---|
| Доставка | logistics |
| Качество товара | product_quality |
| Оплата | payment |
| Возврат | returns |
| Поддержка | support |
| Сайт / приложение | digital |
| Другое | general |

Не менять backend API, если можно обойтись frontend mapping.

---

### `/review/status/:reviewId`

Переделать страницу статуса под клиентский сайт.

Требования:

- ClientLayout;
- понятные статусы для клиента;
- не показывать draft;
- показывать финальный ответ только после публикации;
- показать timeline:
  - Обращение получено;
  - Анализируем отзыв;
  - Ответ на проверке;
  - Ответ опубликован.

Технические внутренние статусы можно преобразовать в человеко-понятные подписи.
